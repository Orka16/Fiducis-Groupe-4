# Sauvegarde de l'AD (srv-ad-bdx) vers Azure — Terraform

Deux machines interviennent :

| Machine | Rôle |
|---|---|
| Ton poste Windows 11 | Lance `az` + `terraform`, crée la cible Azure |
| srv-ad-bdx (192.168.10.1) | Produit le System State et l'envoie vers le blob |

---

## 1. Installer les outils (poste Windows 11)

Dans un terminal **Administrateur** (clic droit sur Démarrer → Terminal (Admin)) :

```powershell
winget install --exact --id Microsoft.AzureCLI
winget install --exact --id Hashicorp.Terraform
```

Ferme puis rouvre le terminal, et vérifie :

```powershell
az version          # attendu : 2.87.x ou plus
terraform version   # attendu : 1.15.x ou plus
```

## 2. Se connecter à Azure

```powershell
az login            # ouvre le navigateur -> virgile.dubernet@ynov.com
az account show     # vérifie l'abonnement actif
az account set --subscription "<ID_ABONNEMENT>"   # si plusieurs
```

Récupère l'ID d'abonnement pour Terraform :

```powershell
az account show --query id -o tsv
```

> Terraform réutilise automatiquement la session `az login` : pas besoin
> de Service Principal pour ce cas simple.

## 3. Déployer la cible Azure (Terraform)

```powershell
cd ad-backup-azure
copy terraform.tfvars.example terraform.tfvars
# édite terraform.tfvars et colle ton subscription_id

terraform init
terraform plan
terraform apply
```

Note le nom du compte de stockage affiché en sortie
(`storage_account_name`), tu en as besoin à l'étape suivante.

## 4. Sauvegarder l'AD (sur srv-ad-bdx)

Sur le DC, installe Azure CLI et connecte-toi :

```powershell
winget install --exact --id Microsoft.AzureCLI
az login
```

Copie `backup-ad.ps1` sur le DC, puis lance-le **en Administrateur** :

```powershell
.\backup-ad.ps1 -StorageAccount "stadbkpbdxXXXXXX" -BackupVolume "D:"
```

Le script :
1. installe Windows Server Backup si absent,
2. fait `wbadmin start systemstatebackup` vers `D:`,
3. envoie le dossier `WindowsImageBackup` dans le conteneur, sous un
   préfixe horodaté.

## 5. Planifier (sauvegarde quotidienne)

Sur le DC, en Administrateur :

```powershell
$action  = New-ScheduledTaskAction -Execute "powershell.exe" `
  -Argument '-NoProfile -ExecutionPolicy Bypass -File C:\scripts\backup-ad.ps1 -StorageAccount stadbkpbdxXXXXXX'
$trigger = New-ScheduledTaskTrigger -Daily -At 2am
Register-ScheduledTask -TaskName "Sauvegarde AD vers Azure" `
  -Action $action -Trigger $trigger -RunLevel Highest -User "SYSTEM"
```

---

## Variante recommandée pour les gros volumes : azcopy

`az storage blob upload-batch` est simple mais lent sur plusieurs Go.
Pour de vraies sauvegardes, installe azcopy et remplace l'étape 3 du
script par une connexion Azure AD (sans clé de compte) :

```powershell
winget install --exact --id Microsoft.Azure.AzCopy.10
azcopy login                     # compte Entra ID
azcopy copy "D:\WindowsImageBackup" `
  "https://stadbkpbdxXXXXXX.blob.core.windows.net/ad-systemstate/$Stamp" `
  --recursive
```

Donne au compte le rôle **Storage Blob Data Contributor** sur le compte
de stockage (Portail Azure → Compte de stockage → Contrôle d'accès (IAM)).

---

## Points d'attention

- **Abonnement Azure** : le compte ynov.com doit avoir un abonnement actif
  (Azure for Students ou autre). Sans abonnement, `terraform apply` échoue.
- **Tombstone lifetime (180 j)** : ne restaure jamais un System State plus
  ancien que 180 jours, l'AD le refuserait.
- **Restauration** : se fait en mode DSRM (Directory Services Restore Mode),
  restauration *non autoritative* par défaut.
- **État Terraform** : `terraform.tfstate` contient les clés du compte de
  stockage. Ne le commite pas dans Git. Pour du sérieux, configure un
  backend distant `azurerm` (un autre compte de stockage dédié à l'état).
- **Réplica La Rochelle** : le même script fonctionne sur srv-ad-lr
  (192.168.20.1) ; pointe-le vers le même conteneur ou un second.
- **Réseau** : le DC doit joindre `*.blob.core.windows.net` en HTTPS (443)
  via le NAT/VPN. Tu peux ensuite restreindre le pare-feu du compte de
  stockage à ton IP WAN (10.10.0.11 côté public).
